# 任务间失败处理机制

## 概述

框架通过 DAG 定义任务间的激活依赖（上游 DONE → 下游 PENDING）。但实际执行中，下游任务可能因为上游服务的实现问题而失败（测试不通过、API 调用失败等）。

此时需要通知上游服务修复，修复完成后重试验证。

**核心机制**：Message Bus 承担任务间通信，框架不限制具体的修复方式。

## 问题场景

```
design → backend → test → deploy
          ↓           ↑
       调用API      验证失败
```

前端调用后端 API 发现缺失或 bug，后端需要修复后前端再重试。

## 解决方案

任务 A 执行失败后，通过 Message Bus 向相关服务发送消息，服务处理完成后任务 A 重试。

```
任务 A 执行失败
    ↓
识别需要通知的服务
    ↓
通过 Message Bus 发送消息
    ↓
服务接收消息，修复问题
    ↓
任务 A 轮询消息状态
    ↓
所有修复完成后，任务 A 重试
```

## 框架职责

| 组件 | 职责 |
|------|------|
| Aggregator | DAG 调度（依赖 DONE → 下游 PENDING） |
| Message Bus | 任务间消息传递和状态追踪 |
| Agent | 自行决定发什么消息、怎么修复、重试几次 |

**框架不做限制**：
- 失败后应该通知谁
- 修复的具体方式
- 重试的次数和间隔

## 与 DAG 依赖的区别

| | DAG 依赖 | 消息通信 |
|---|---|---|
| 阶段 | 任务激活前 | 任务执行中 |
| 方向 | 上游 → 下游 | 任意方向 |
| 触发 | 依赖 DONE | 任务失败 |
| 状态 | 任务状态 | 消息状态 |

DAG 定义了"谁先谁后"，Message Bus 处理"谁出了问题谁去修"。
